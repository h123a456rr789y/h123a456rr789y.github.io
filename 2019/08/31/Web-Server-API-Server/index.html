Web Server & API Server

Posted on 2019-08-31 Edited on 2022-08-11

Web Server and API Server

Web Container

  • Web容器是一種服務調用的規範，其中有Web Container和EJB Container等
  • Web容器給處於其中的應用程序組件（JSP、Servlet）提供環境，JSP、Servlet直接與容器中的環境變量打交道，而不必關注其他系統問題
  • Web容器更多的是與HTTP的請求打交道，而EJB容器不是，它更多的是跟數據庫其他服務打交道

Web Server

  • Web服務器可以處理HTTP協議，當Web服務器收到一個HTTP請求，就會返回一個HTTP響應。
  • Apache Server:是簡單的HTTP服務器，速度快，在大型網站中，常與應用服務器配合部署，僅支持靜態頁面。
  • Apache Nginx: 著名的HTTP服務器，速度快，支持負載均衡。
  • client-server 架構:
    • Client端可以想像成是一個主動的角色，它會發送請求(request)給server端，並開始等待直到收到回應。
    • Server端可以想像成是一個被動的角色，它會等到client端發送請求(request)來後開始處理，並回傳一個回應(response)給client端。
  • P2P架構:
    • A分享一個檔案上去，經hash function 後產生一組key，client就可透過這個key 來access 到A分享的檔案

Aplication Server

  • 作為應用服務器，要求可以通過各種協議（包括HTTP協議）把商業邏輯暴露給客戶端應用程序，扮演一個分工管理的腳色
  • Tomcat: Tomcat僅是個HTTP服務器，是Apache鼎力支持的Java Web服務器，是Servlet/JSP API的執行器，也就是所謂的Servlet容器，不支持EJB、JMS。相對於JBOSS來說較為輕量級，實現的功能也較少，不能完整的支持最新的JavaEE標準，適用於較為簡單的Web應用
  • JBOSS: JBOSS是開源Java EE應用服務器，它不但是Servlet容器，而且是EJB容器，它提供了更多開箱即用的功能特性，也更重，它完整的實現了J2EE 6。
  • Glassfish: Glassfish是Oracle 開發的官方Java EE容器，也是同時支持Servlet和EJB，支持最新的特性，有自己的web容器，支持集群，支持熱部署。

Restful Web API

Why we need Restful API

  • The cons of traditional web service:

    • Data and presentation logic are bund together, which make it difficult to debug and test.
    • Either the change of Data or the presentation logic will have cause the other to change as well. Thus the cache system doesn’t work
    • Intergration of Data and presentation logic will bring the heavy load for servers

  • 所以將Data跟Presentation Logic徹底切開，Server一分為二成前端Web跟後端API

  • Pros:

    • Web Server可以透過快取機制將資料存在Web Server端，如果資料沒有改，就只須執行 presentation logic 的部分 就可以傳給client了，有效的降低server負擔
    • Client端也有他的快取機制，只要前端的設計沒有改過，presentation logic不變的情況下，client端也沒必要發重發request給Web Server，Presentation不用重傳，節省網路流量
    • 開發方面也能穩定許多，前後端邏輯職責劃分清楚不互相影響，並且在後期開發上，Api Server也還能重複給不同的平台使用，提高系統的擴張性

  • Restful API(Representational State Transfer):是一種API的設計風格，表徵狀態的切換，表徵指的就是前端網路介面(HTML)，狀態切換是指使用者如何與表徵互動，當表徵發現改變，Web server就要和API server 互動

  • Restful API pros:

    • 簡單且統一的Api接口，讓不同的前端Server能夠更方便的與後端Api對接，並且每一種url接口就代表一種data，降低了系統資料之間的耦合性，讓後端api開發上好維護且彈性許多

  • Resful API cons:

    • 低耦合就代表著高鬆散，當前端Web的Presentation Logic過於複雜時，例如一個頁面(表徵)需要多種資料時，就必須多發幾個request向後端要資料，例如一個部落格頁面除了要文章內容還要作者資料時，就必須一次發兩個request，一旦Presentation Logic複雜起來，就會大大影響系統的效能。
